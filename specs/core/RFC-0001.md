# CAP-Core RFC-0001: Machine-Operable Object Assembly Contract

> Status: draft proposal - Non-normative - Last updated: 2026-07-05

This RFC is the first integrated CAP-Core proposal. It is not a stable
specification and does not define CAP-Core conformance.

> Split note: as of 2026-07-06, this integrated draft is retained as an overview
> while focused split drafts are prepared in
> `RFC-0001-core-object-model.md`, `RFC-0002-binding-model.md`,
> `RFC-0003-scientific-computation-profile.md`, and
> `RFC-0004-cap-digest-bridge-profile.md`.

## 1. Abstract

CAP-Core is proposed as a cross-layer assembly contract for machine-operable
agent tasks. It describes artifacts, capabilities, runtime/resource/service
bindings, policy bindings, assemblies, runs, and run evidence. CAP-Core should
bind to existing standards rather than replace them. CAP-Digest remains the
context evidence profile for model-visible views.

## 2. Status of This Document

CAP-Core was previously reserved. This RFC starts a draft specification track,
but only as a proposal. Implementers should treat every object and field in this
document as provisional until a future CAPP accepts schemas, fixtures, and
conformance tests.

## 3. Motivation and Problem Statement

Agent tasks often cross boundaries that are specified by different systems:

- data and code live in files, repositories, object stores, databases, or
  research-object packages;
- operations are exposed through tools, services, skills, workflows, or local
  host APIs;
- execution happens through local processes, containers, WebAssembly, clusters,
  remote execution services, or workflow engines;
- policy decisions come from identity systems, policy engines, host rules, and
  user consent;
- evidence appears as logs, traces, outputs, provenance records, attestations,
  signatures, dependency inventories, or model-visible digests.

Each system can be useful and mature, but no reviewed source provides a compact
contract that connects all of these parts for review before execution and audit
after execution. CAP-Core fills that control-plane gap.

CAP-Core does not make an agent smarter. It makes the selected objects,
bindings, decisions, runs, and evidence explicit enough for machines and people
to inspect.

## 4. Scope and Non-Goals

CAP-Core should include:

- identity and references for artifacts;
- capability declarations at the contract level;
- typed bindings to runtime, resource, service, policy, transport, data-plane,
  schema, evidence, and digest systems;
- assembly and run lifecycle records;
- profile and binding extension points;
- minimal policy decision and evidence envelopes.

CAP-Core should exclude:

- transport protocols;
- full agent runtimes;
- workflow languages;
- container, sandbox, or scheduler formats;
- policy languages;
- identity protocols;
- secret management;
- SBOM or attestation formats;
- full provenance ontologies;
- model reasoning guarantees;
- CAP-Digest field selection and digest text details.

Boundary decisions are tracked in `BOUNDARY-MATRIX.md`.

## 5. Relationship to CAP-Digest

CAP-Digest and CAP-Core are separate layers:

```text
CAP-Core
  Artifact graph
  Capability contract
  Runtime / resource / service binding
  Policy binding
  Assembly
  Run
  RunEvidence
    |
    | may reference or render
    v
CAP-Digest
  SourceRef
  Field Catalog
  Digest
  DigestManifest
  DigestEvidence
```

Rules:

- CAP-Digest may render a CAP-Core Artifact, Assembly, Run, or RunEvidence
  object into model-visible context.
- CAP-Core must not depend on digest text as its only evidence format.
- DigestEvidence and RunEvidence are distinct.
- A DigestBinding can connect a RunEvidence record to a CAP-Digest view, but the
  digest view does not replace run evidence.

## 6. Terminology

| Term | Draft meaning |
|---|---|
| `Artifact` | Data, code, model, config, result, log, image, environment reference, document, or other machine-operable object. |
| `ArtifactRef` | Stable, scoped, URI-addressed, content-addressed, or profile-defined reference to an Artifact. |
| `Profile` | Domain or execution-specific interpretation of Core objects. |
| `Capability` | Declared operation that can act on artifacts under constraints. |
| `Binding` | Typed edge from a Core object to an external standard, host object, runtime, service, policy decision, evidence record, or digest view. |
| `Assembly` | Pre-run contract connecting artifacts, capability, profiles, bindings, and policy. |
| `Run` | One execution or observation instance created from an Assembly. |
| `RunEvidence` | Evidence envelope containing or referencing records from a Run. |
| `PolicyBinding` | Binding to an authorization or policy decision system plus minimal decision record. |
| `DigestBinding` | Binding from a Core object or RunEvidence record to a CAP-Digest view. |

Terminology separation is reviewed in `PRIMITIVE-REUSE.md`.

## 7. Layer Model

CAP-Core should sit between external systems and CAP-Digest:

```text
External standards and hosts
  MCP, A2A, Skills, CWL, WDL, Nextflow, Snakemake, RO-Crate,
  OCI, WASI, Kubernetes, Slurm, REAPI, W3C PROV, in-toto,
  Sigstore, SLSA, SPDX, CycloneDX, OPA, Cedar, OpenTelemetry,
  Arrow Flight, JSON Schema, JSON-LD, SHACL

CAP-Core
  ArtifactRef, Capability, Binding, Assembly, Run, RunEvidence

Profiles and bindings
  domain profiles, workflow profiles, runtime bindings, evidence bindings,
  policy bindings, data-plane bindings, digest bindings

CAP-Digest
  model-visible context evidence views
```

CAP-Core should remain a control plane. Data movement, execution, policy
evaluation, and telemetry collection belong to the external systems that already
own those functions.

## 8. Core Object Model

### Artifact

An Artifact is a machine-operable object that can participate in an Assembly.

Candidate fields:

```text
id
kind
mediaType
profileType
identity
version
digest
labels
metadata
profiles
bindings
```

### ArtifactRef

An ArtifactRef identifies or locates an Artifact without requiring Core to move
the bytes.

Candidate fields:

```text
ref
scope
kind
integrity
profile
accessHints
```

### Profile

A Profile constrains interpretation of a Core object. Core defines the reference
mechanism; the profile owns domain semantics.

Candidate fields:

```text
id
name
version
appliesTo
schemaRef
constraints
compatibility
```

### Capability

A Capability is a declared operation that may act on artifacts under explicit
constraints. It is not the same thing as a tool call.

Candidate fields:

```text
id
name
description
inputs
outputs
sideEffects
requiredBindings
errorSemantics
profiles
implementationBindings
```

### Binding

A Binding connects Core objects to external systems.

Candidate fields:

```text
id
type
target
standard
standardVersion
profile
constraints
integrity
freshness
status
evidenceRefs
```

The first draft uses binding roles rather than separate required top-level
objects for every binding type.

### Assembly

An Assembly is the central CAP-Core object. It says what is assembled before any
run starts.

Candidate fields:

```text
id
artifacts
capability
profiles
bindings
policyBinding
state
createdAt
createdBy
unresolved
```

### Run

A Run is an instance created from an Assembly.

Candidate fields:

```text
id
assemblyId
state
startedAt
endedAt
inputs
outputs
runtimeBinding
resourceBinding
serviceBindings
logs
evidenceRefs
```

### RunEvidence

RunEvidence is an envelope for records produced by or about a Run.

Candidate fields:

```text
id
runId
evidenceType
profile
subject
materials
products
logs
records
attestations
provenanceRefs
sbomRefs
telemetryRefs
digestViews
completeness
integrity
```

The full primitive review appears in `PRIMITIVE-REUSE.md`.

## 9. Assembly Lifecycle

CAP-Core should record lifecycle checkpoints rather than define an execution
engine.

Draft lifecycle:

```text
discover -> describe -> assemble -> bind -> authorize -> record assembly
  -> start run -> observe -> finalize -> record evidence -> render digest view
```

Checkpoint meanings:

| Stage | Draft purpose |
|---|---|
| discover | Find candidate artifacts, capabilities, profiles, and bindings. |
| describe | Normalize enough metadata to decide whether an object can participate. |
| assemble | Create a candidate Assembly. |
| bind | Resolve runtime, resource, service, policy, evidence, transport, digest, data-plane, and schema references. |
| authorize | Obtain and record policy decision. |
| record assembly | Persist or return the reviewable pre-run contract. |
| start run | Create a Run from an Assembly. |
| observe | Attach status, logs, metrics, partial outputs, and policy events. |
| finalize | Close the Run and record final status. |
| record evidence | Produce RunEvidence or evidence bindings. |
| render digest view | Optionally expose a CAP-Digest view. |

The first RFC does not require a strict state machine. Candidate run states are
`planned`, `starting`, `running`, `waiting`, `completed`, `failed`,
`cancelled`, and `stale`.

## 10. External Standard Binding Policy

CAP-Core bindings should identify external standards and host systems without
copying them. The binding policy is defined in `BINDING-POLICY.md`.

Summary:

| Area | Reuse candidate | CAP-Core behavior |
|---|---|---|
| Agent connection | MCP, A2A | Bind or reference; do not replace. |
| Procedure packaging | Codex Skills, Anthropic Skills, Copilot instructions | Reference as procedural guidance; do not treat as executable contract. |
| Workflow | CWL, WDL, Nextflow, Snakemake | Profile or binding; do not define workflow language. |
| Research object | RO-Crate, Workflow Run RO-Crate, BagIt | Archive/profile binding; do not replace packaging. |
| Runtime | OCI, WASI, Kubernetes, Slurm, REAPI | Runtime binding; do not define runtime or scheduler. |
| Policy | OPA, Cedar, OAuth/OIDC, SPIFFE/SPIRE | Policy binding; do not define policy language or identity protocol. |
| Provenance | W3C PROV, Workflow Run RO-Crate, MLflow, DataLad | Evidence binding; do not invent full provenance model. |
| Supply chain | in-toto, SLSA, Sigstore, SPDX, CycloneDX | Evidence binding; do not define signing, attestation, or SBOM format. |
| Observability | OpenTelemetry, host logs | Evidence input; do not define telemetry protocol. |
| Data plane | Arrow Flight, object stores, filesystems | Reference handles; do not move large bytes through Core. |
| Schema | JSON Schema, JSON-LD, SHACL, CodeMeta | Profile or validation binding; do not require one semantic stack. |

## 11. Security and Policy Model

CAP-Core should define security records, not enforcement engines.

The first draft policy model uses:

```text
principal
action
resource
conditions
decision
decisionTime
policyRef
auditRef
```

Candidate decision values:

```text
allowed
denied
allowed_with_constraints
needs_confirmation
stale_context
```

Secrets must be represented by reference only. Network, data locality, write
scope, resource, and cost constraints should be visible before a Run starts.
Denied and stale decisions should be first-class records.

Security details are in `SECURITY-POLICY-DRAFT.md`.

## 12. Profiles

Profiles carry semantics that do not belong in Core. Candidate initial profiles:

- scientific computation profile;
- ML experiment profile;
- batch workflow profile;
- notebook analysis profile;
- data validation profile;
- software build/test profile;
- CAP-Digest view profile.

Core should define how to reference a profile. The profile defines domain fields,
validation, compatibility, and optional external mappings.

## 13. Conformance Strategy

Stable conformance is not accepted by this RFC. The draft strategy is:

| Level | Name | Capability |
|---:|---|---|
| 0 | Core Object Reader | Parse and preserve Core records and profile extensions. |
| 1 | Assembly Producer | Emit a complete pre-run Assembly. |
| 2 | Run Recorder | Record Run and RunEvidence linked to an Assembly. |
| 3 | Binding Ecosystem | Validate at least one external standard binding and one CAP-Digest view. |

The first schema sketches use JSON Schema under `schemas/core/`. Optional
JSON-LD or SHACL profiles can follow after the Core object shape stabilizes.

The first fixture family is:

```text
fixtures/core/local-analysis/
```

The first reference prototype is a validator/renderer under
`reference/python/cap_core/`. Conformance planning is in
`CONFORMANCE-DRAFT.md`. These assets are draft-track evidence, not stable
CAP-Core conformance.

## 14. Example Assembly

The first example is local scientific analysis:

```text
dataset + script + environment reference + config
  -> capability: run-analysis
  -> runtime binding: local process or OCI reference
  -> resource binding: CPU/memory/time/network/storage
  -> policy binding: user/session authorization and no network
  -> run
  -> run evidence
  -> optional CAP-Digest view
```

This example validates that CAP-Core can describe artifact references,
capability intent, runtime/resource/policy bindings, a run instance, evidence
records, and an optional digest view without defining a workflow engine, runtime,
policy language, provenance ontology, or agent transport protocol.

The example lives in `EXAMPLE-ASSEMBLY-0001.md`.

## 15. Compatibility and Versioning

The first draft should use explicit versions for:

- Core schema versions;
- profile versions;
- binding versions;
- external standard versions or reviewed dates;
- CAP-Digest version compatibility.

Core records should preserve unknown profile fields where possible. Breaking
schema changes should require a future CAPP and fixture updates.

## 16. Open Questions and Deferrals

Open questions are tracked in `OPEN-QUESTIONS.md`. The current dispositions are
only sufficient to draft RFC-0001. They do not close the design.

Major deferrals:

- strict run state machine;
- central external-standard registry;
- mandatory JSON-LD representation;
- mandatory W3C PROV representation;
- separate top-level objects for every binding type;
- reference runtime implementation;
- normative conformance levels.

## 17. Source Basis

The baseline source review is in `RESEARCH-SOURCES.md`. RFC-0001 intentionally
separates:

- established standards;
- observed implementation practice;
- CAP design hypotheses;
- unresolved research questions.

The draft source conclusion is that CAP-Core should be a machine-operable object
assembly contract, not a transport protocol, workflow language, runtime,
provenance standard, policy engine, SBOM format, or CAP-Digest expansion.
